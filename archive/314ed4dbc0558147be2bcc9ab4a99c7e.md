## Notion MCP を使ってみた感想
Notion MCP（Model Context Protocol）を使ってみて、以下のような点を感じました。
- **認証がシンプル**  
	`mcp_auth` を一度実行するだけで、社内の Notion ワークスペースに接続でき、以降は MCP 経由でページ作成・検索・取得ができる。
- **URL 指定で直感的**  
	データベースやページの URL を渡すと、`notion-fetch` でスキーマやデータソース ID を取得でき、そのまま `notion-create-pages` で記事やページを追加できる。どの DB に何を追加するかが分かりやすい。
- **データベースとページの扱いの違いが明確**  
	はじめは「ページ」だと思っていた URL が実はデータベースだったりするが、fetch で type や data-source が分かるので、`page_id` と `data_source_id` を使い分けて正しく作成できる。
- **記事・ドキュメントの一元管理に便利**  
	ms_news のような DB に「名前」と本文（content）で記事を積み重ねていけるので、ニュースやメモを Notion に集約する運用と相性が良い。
今後は、既存ページの更新（`notion-update-page`）や検索（`notion-search`）も組み合わせて、下書き→公開の流れやタグ付けまで MCP でできるとさらに良さそうだと感じました。

## Skills がどういう役割をしているか
**Skills** は、Cursor のエージェントが「いつ何をするか」を判断するための**ガイド（役割・手順の説明）**です。
- **トリガーとなる言い回しの定義**  
	各 Skill には `description` があり、「Notion に検索して」「ページを作成して」「調査してドキュメント化して」などの**ユーザーの言い回し**に反応して、その Skill が選ばれやすくなる。
- **ワークフローの指示**  
	Skill の本文（[SKILL.md](http://SKILL.md)）には「まず検索 → 次に fetch → そのあと create」といった**手順**が書かれており、エージェントが Notion MCP のどのツールをどんな順で使うべきかが分かる。
- **MCP ツールとの役割分担**  
	- **MCP**: 実際の API 呼び出し（`notion-create-pages`, `notion-fetch` など）を行う**実行役**。  
	- **Skills**: ユーザーの意図を解釈し、どの MCP ツールをどう使うかを決める**設計図・手順書**。
- **コンテキストの付与**  
	「会議メモなら Attendees / Agenda / Action items を入れる」「調査レポートなら Executive summary → Findings → Citations で書く」といった**ドメインごとの型**を Skill が持たせることで、単に API を叩くだけでなく、適切な形で Notion に書き出せる。
まとめると、**MCP が「できること」を提供し、Skills が「いつ・どう使うか」をエージェントに教える役割**をしています。そのため、同じ Notion MCP でも、create-page 用の Skill と research-documentation 用の Skill では、ユーザーの依頼の受け方と実行の流れが変わります。
